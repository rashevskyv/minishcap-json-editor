# Plan: Fix Warning Display & System Analysis

## Current Status
The application launches successfully, but **warnings (e.g., width exceeded, short lines) are not visible** in the editor or the string list, despite the underlying logic seemingly detecting them (verified by tests).

## Warning System Architecture (My Understanding)

The warning system operates in a pipeline:

1.  **Detection (`IssueScanHandler`)**:
    *   Scans blocks/strings using `ProblemAnalyzer` (from plugins).
    *   Calculates widths using font maps.
    *   Populates `self.mw.problems_per_subline` dictionary.
    *   **Key Format**: `(block_idx, string_idx, subline_idx)`
    *   **Value**: `Set[problem_id]`

2.  **Storage (`MainWindow`)**:
    *   `problems_per_subline`: Central store for all detected issues.
    *   `detection_enabled`: Configuration dict mapping `problem_id` to `bool`.

3.  **Visualization - Editor (`UIUpdater` -> `TextHighlightManager`)**:
    *   `UIUpdater._apply_highlights_to_editor` is called on text changes.
    *   It iterates through editor lines (sublines).
    *   Lookups problems in `problems_per_subline` using `(block_idx, string_idx, editor_line_idx)`.
    *   Calls `highlightManager.addWarningLineHighlight` (or Critical).
    *   `TextHighlightManager` uses `QTextEdit.ExtraSelection` to paint background colors.

4.  **Visualization - Line Number Area (`LNETLineNumberAreaPaintLogic`)**:
    *   Paints the sidebar (line numbers).
    *   Lookups problems using the same key format.
    *   Draws colored bars/icons if problems exist.

5.  **Visualization - String List (`UIUpdater.populate_blocks`)**:
    *   Aggregates problems for the whole block.
    *   Updates the list item text (e.g., "Block 0 (3 Width)").

## Hypotheses for "Missing Warnings"

1.  **Data Key Mismatch**: The `subline_idx` generated by `IssueScanHandler` might not match the `blockNumber()` in the editor.
    *   *Check*: Are we handling multi-line strings correctly in both places?
2.  **Rendering Override**: `LNETPaintEventLogic` (custom paint event) might be drawing *over* the standard `ExtraSelections`, hiding the highlights.
3.  **Configuration**: `detection_enabled` might be defaulting to `False` or missing keys.
4.  **Timing**: Highlights might be applied *before* the text is fully loaded or layout is calculated, causing them to be lost.
5.  **Font Map/Width Calculation**: If the font map is missing or incorrect in the live app, `calculate_string_width` might return 0 or a small value, resulting in no "Width Exceeded" problems being detected in the first place.
